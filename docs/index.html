Coverage

What it checks.

AWS-first in v0.1, with resource-specific recoverability rules for destructive infrastructure changes. The goal is not generic “risk scoring”; it is a defensible answer to: can we get this back?

01 RDS deletion protection, final snapshots, backup retention
02 S3 versioning, MFA delete, object lock, replication
03 DynamoDB PITR, deletion protection, table backups
04 KMS deletion windows and dependent encrypted resources
05 EBS snapshots, AMI backing, volume deletion behavior
06 CloudWatch Logs retention, exports, loss windows
07 IAM role, policy, and access-dependency impact
08 + more EC2, Lambda, VPC, ELB, Route53, SNS, SQS
Design principles

Checkable verdicts, not vibes.

No inference theater
It does not guess that something “looks like” production. It reads explicit values in your plan and reports the recoverability consequence.
No severity-by-vibes
Verdicts are factual: recoverable, recoverable-with-effort, recoverable-from-backup, unrecoverable, or blocked.
No moralizing your config
It does not merely say “set deletion_protection.” It tells you what would change the recoverability verdict if you did.
No agents or SaaS
It runs as a CLI on your machine or in CI. No account, no external API, no plan data leaving the process.
Honest limits

What v0.1 does not cover.

Terraform plans only. Not Pulumi, CloudFormation, or raw API calls yet.

AWS-first. GCP and Azure are on the roadmap. The analysis framework can expand, but provider-specific resource libraries take time.

Plan-file analysis, no live state. If a backup vault exists outside your Terraform configuration, RecourseOS cannot see it. --live-state is in development.

Static analysis, not approvals. v0.1 flags and exits non-zero in CI on unrecoverable changes. Approval workflows come later, after the labels have earned trust.
