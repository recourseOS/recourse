Why preflight?

It is the check you run before the risky action runs.

preflight is not a separate install. It is the Recourse command that evaluates a proposed action while there is still time to stop, change, or approve it.

before
Catches destructive intent early
Run it before `terraform apply`, a shell delete, or an agent MCP call.
decision
Gives an operator verdict
Outputs run, warn, review, or block instead of leaving people to interpret raw diffs.
evidence
Explains what is missing
Shows the detected target, recovery evidence found, and evidence needed for approval.
agents
Works as an agent safety layer
The same evaluator can be called through MCP before an agent touches infrastructure.
Platform

One evaluator. Multiple action surfaces.

Agents do not only edit Terraform. They run shell commands, invoke MCP tools, and call provider APIs through wrappers. RecourseOS normalizes those proposed mutations into the same consequence model.

01
Terraform plan analysis
Reads plan JSON and classifies destructive changes by recoverability, evidence, and decision.
02
Shell mutation preflight
Detects destructive command patterns and escalates when live recovery evidence is missing.
03
MCP tool-call guard
Exposes a local MCP server so agents can ask for a verdict before invoking risky tools.
04
BitNet-ready unknowns
Known rules remain first. Unknown resources route through conservative semantic classification.
Why not just plan?

Terraform shows the diff. Recourse shows the point of no return.

A plan can tell you a database will be destroyed. Recourse tells you whether that destruction is reversible, recoverable from backup, unrecoverable, or too ambiguous for an agent to treat as safe.

diff vs consequence

- aws_db_instance.main will be destroyed

Coverage

Rules where safety is knowable. Review where it is not.

Provider-specific handlers cover common AWS, GCP, Azure, and Azure AD resources. Unknown resources use semantic safety signals and stay conservative when evidence is thin.

AWS
Data, identity, network
RDS, DynamoDB, S3, EBS, IAM, KMS, Secrets Manager, VPC, Route53, Lambda, queues, and more.
GCP
Storage, SQL, IAM
GCS versioning, Cloud SQL protection and backups, Secret Manager, BigQuery, IAM bindings, and long-tail review paths.
Azure
Storage, database, identity
Storage soft delete, SQL family protection signals, Cosmos DB, Key Vault, role assignments, and Azure AD resources.
Unknown
Semantic classifier
Deletion protection, backup windows, versioning, retention, and force-destroy signals are evaluated without hardcoding every resource.
Output

Structured reports for agents, readable reports for humans.

The schema is designed for machine consumption: enum-like decisions, recoverability tiers, confidence, evidence, and missing evidence. The CLI keeps that same report inspectable in CI.

decision
block
Stop an action when evidence shows an unrecoverable consequence.
tier
backup
Distinguish reversible config changes from backup-dependent recovery.

review
Escalate unknowns instead of guessing that a destructive change is safe.
